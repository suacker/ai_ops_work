# BI 网站改版思考：参考 BI4Sight 的增长工作台叙事

> 日期：2026-06-02  
> 参考站点：https://www.bi4sight.com/zh/Ai  
> 适用仓库：`/Users/suacker/dev/DataAnalysis/BI`

## 1. BI4Sight 网站阅读结论

BI4Sight 的官网不是单纯展示“BI 报表能力”，而是在包装一个“AI + BI + 投放执行”的完整增长工作台。它的主叙事是：让每一分预算得到确定性增长，并把产品能力拆成用户能理解的工作节点。

核心导航包括：

- 用 AI：AI 盯盘、AI 止损、定时策略、异常预警、Meta 建议采纳、策略模板沉淀。
- 发广告：Meta/Google/TikTok 批量发布、叉乘测试、跨账户同步、定时策略、批量优化。
- 看数据：多平台数据整合、Meta 官方诊断、增长机会、多维下钻、行业对标、自定义报表、空耗管控。
- 管素材：素材库、合规校验、爆款拆解、生命周期、跨角色协作、素材报表。
- 管风险：防盗刷、违规预警、账户健康、预算/落地页防护、风险复盘。
- 管资产：账户、授权、像素、域名等投放资产统一健康管理。
- 归因：Web2App / 多渠道效果归因。
- API：连接内部系统和媒体生态。

行业解决方案按短剧、网文、游戏、AI 工具、跨境电商、社交 APP 拆分，每个行业页都采用“痛点 -> 场景 -> 功能组合 -> 结果指标 -> 案例”的模板。这个结构很适合转化，但不适合直接照搬到我们的内部 BI，因为我们的核心不是销售线索，而是日常决策效率。

## 2. 对我们当前 BI 的关键启发

我们现有体系已经有很强的方法论基础：`BI系统报告设计.md` 定义了报表全景、Cohort First、国家优先、阈值驱动、MD+HTML 双出；`specs/UI_design.md` 已经定义了顶部 Tab + 左侧菜单 + 报表内筛选器；`grow_base/` 和 `monitor_report/grow/` 已经能跑出国家阈值和 5 天监控报告。

但现在的表达更像“报表目录”和“静态报告集合”，缺少一个清晰的“增长作战工作台”入口。参考 BI4Sight 后，改版的重点不是换视觉，而是把用户任务重新组织：

- 从“我有哪些报表”改成“今天哪里有问题、我该做什么、证据是什么”。
- 从“看数据”升级为“发现异常 -> 归因定位 -> 生成动作 -> 追踪结果”。
- 从“HTML 报告归档”升级为“报告即证据链，可从看板一键下钻”。
- 从“阈值表是文件”升级为“阈值是每个决策卡片旁边的上下文”。

## 3. 改版定位

建议把我们的 BI 网站定位为：

**IAA/IAP 增长运营决策台**

副标题：

**从 Snowball 数据到国家 / 渠道 / Campaign 动作建议的一站式增长工作台。**

不要使用营销官网式的大 Hero。我们的用户是内部增长、买量、产品、变现、数据团队，第一屏应该直接进入工作台，而不是品牌宣传页。

## 4. 信息架构建议

沿用现有 `specs/UI_design.md` 的顶部 Tab + 左侧菜单，但把模块命名做得更贴近工作流：

| 顶部模块 | 目的 | 首屏默认 |
|---|---|---|
| 作战台 | 今天是否安全，哪里要处理 | 全局健康 + P0/P1 动作队列 |
| 分析报告 | 周期性深度分析与历史报告 | 最近生成的报告列表 |
| 基线与阈值 | grow_base、阈值漂移、国家策略 | 项目国家阈值表 |
| 实验 | A/B 预注册、监控、结论 | 实验列表 |
| 设置 | 数据源、通知、权限、默认首页 | 数据源状态 |

左侧菜单建议按“决策场景”而不是报表编号排序：

- 总览：全局健康、项目健康、数据完整性。
- 增长：国家放量优先级、5 天国家监控、渠道健康、Campaign 质量、创意效率。
- 变现：eCPM 趋势、广告频次、IAP 漏斗、订阅续费。
- 回收：ROAS Dn、回正日预测、月度回收复盘。
- 风险：阈值突破、留存崩塌、版本影响、预算异常。

报表编号（EX-01、MN-01、UA-02）保留在页面副标题或徽章中，不作为用户主要导航语言。

## 5. 第一屏设计

第一屏应该回答 4 个问题：

1. 今天整体安全吗？
2. 哪些国家 / 渠道 / Campaign 需要立即处理？
3. 哪些地方可以加预算？
4. 每条建议的证据和阈值是什么？

推荐布局：

```text
顶部栏：项目选择器 | 日期窗口 | 数据更新时间 | 告警入口 | 用户

左侧：作战台菜单

主内容：
  1. 今日总览条：总花费、日均花费、cohort ROAS、高风险国家数、P0 动作数
  2. P0/P1 动作队列：每条动作带国家、渠道、指标、阈值、建议动作
  3. 国家状态矩阵：国家 x ROI0/CPI/次留/eCPM，按健康/中风险/高风险着色
  4. 增长机会卡：可放量国家、可复用素材/渠道、低 CPI 高 ROAS 组合
  5. 最近报告：MN-01、UA-02、RO-02 等最新产物
```

## 6. 关键页面改版

### 6.1 作战台首页

把 BI4Sight 的“AI 盯盘 / AI 止损 / AI 预警”转成我们的内部决策组件：

- AI 盯盘 -> 阈值突破监控：ROI0、CPI、次留、eCPM、花费突增。
- AI 止损 -> 停投/降量建议队列：只输出数据支持的建议，不自动执行。
- AI 预警 -> P0/P1 风险列表：明确异常类型、触发时间、责任维度。
- 策略模板 -> 国家/渠道动作模板：扩量、控 CPI、修留存、修素材、停投复查。

### 6.2 国家放量优先级页

这是我们最应该产品化的页面。页面结构：

- 顶部：项目、日期、阈值来源、数据置信度。
- 左侧：国家优先级列表，可按“可放量 / 观察 / 降量”筛选。
- 右侧：选中国家的证据面板，包括 ROI0、CPI、次留、eCPM、花费、ROAS D7/D14。
- 底部：动作建议和下钻链接：渠道健康、Campaign 质量、创意效率。

### 6.3 Campaign 质量页

借鉴 BI4Sight 的“发广告”和“看数据”，但我们的重点不做投放创建，而是做投后诊断：

- Campaign 质量分。
- 风险原因拆解：ROI 不达标、CPI 超线、留存低、样本不足。
- 建议动作：扩量、降量、观察、停投、素材替换、等待 cohort 成熟。
- 每个建议必须有阈值证据和数据窗口。

### 6.4 素材效率页

借鉴“管素材”的表达，但先做轻量版：

- 创意 / 素材维度 ROAS、CTR、IPM、CPI。
- 国家与渠道适配情况。
- 爆款复用建议。
- 衰退预警。
- 素材合规与生命周期可以作为 Phase 2。

### 6.5 风险页

把风险从“报表”升级为一个持续入口：

- 数据完整性：Snowball 数据是否缺失、回传是否延迟。
- 阈值突破：项目 x 国家 x 渠道。
- 预算异常：花费突增、花费无转化。
- 留存异常：D1/D3 快速下滑。
- 版本影响：版本发布前后 ROI/留存/eCPM 变化。

## 7. 视觉方向

不要照搬 BI4Sight 的营销视觉。我们需要的是高密度、低装饰、可扫描的运营工具界面。

建议保留当前设计系统的浅色、蓝绿色品牌色和状态色，但做三点调整：

- 减少大面积渐变和装饰背景，作战台用白底、浅灰分区、状态色左边条。
- 卡片更紧凑：主卡片圆角控制在 8px 左右，KPI 卡用 12px 内边距。
- 表格是核心组件：冻结首列、状态徽章、阈值线、异常值着色、行展开证据链。

状态色保持当前规范：

- 健康：绿色。
- 中风险：黄色。
- 高风险：红色。
- 数据缺失/低置信度：灰色。

## 8. MVP 拆分

### Phase 1：把现有报告产品化

目标：不新增太多数据口径，先把已有 `grow_base`、`monitor_report/grow`、`reports/` 变成可浏览的 BI 网站。

- 作战台首页。
- 报告列表页。
- 报告详情页：嵌入现有 HTML 报告。
- 国家阈值页：读取 `grow_base/{project}.md`。
- 5 天监控页：读取 `monitor_report/grow/{project}_{MMDD}.md/html`。
- 全局搜索：项目、国家、报表 ID、报告标题。

### Phase 2：交互式分析

- 国家状态矩阵。
- 国家详情页。
- 渠道健康页。
- Campaign 质量页。
- eCPM 广告位趋势页。
- 动作队列状态：待处理、处理中、已验证、已关闭。

### Phase 3：AI 辅助与自动化

- 异常摘要自动生成。
- 动作建议模板化。
- 阈值漂移提醒。
- 报告自动排程。
- 飞书/邮件推送。

## 9. 与现有仓库的落地点

建议新增：

```text
site/
  app/
  components/
  data/
  report-index.json
```

或者如果先做纯静态版本：

```text
website/
  index.html
  assets/
  reports-index.json
```

先不要把所有报告生成逻辑迁入网站。更稳的路径是：

1. 继续让现有 skill 生成 MD + HTML。
2. 增加一个 report index 生成脚本，扫描 `grow_base/`、`monitor_report/`、`reports/`。
3. 网站读取 index，按项目、日期、类型展示。
4. 交互式页面逐步替换静态 HTML 报告。

## 10. 最重要的产品原则

改版后每个页面都必须坚持：

- 页面第一屏先给结论，不先给图表。
- 每个结论必须能下钻到国家，必要时到渠道/Campaign。
- 每个指标旁边必须有阈值、状态或趋势上下文。
- 每条动作建议必须能说明“为什么”和“看哪份证据”。
- 数据缺失必须显示出来，不能隐藏。
- MD 是真值，HTML/网站只是渲染层。

## 11. 建议的下一步

最小可执行版本建议先做：

1. `reports-index.json` 生成器。
2. 一个静态 BI 首页，读取索引并展示最近报告。
3. 一个作战台原型，用固定 mock 数据还原 MN-01 的国家风险矩阵。
4. 一个报告详情页，直接嵌入现有 HTML 报告。

这能在不重写分析链路的前提下，把“报告文件夹”升级为“可浏览、可搜索、可下钻的增长决策台”。
