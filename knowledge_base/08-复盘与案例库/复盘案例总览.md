# 08 复盘与案例库

## 定位

复盘与案例库用于沉淀成功和失败经验，重点记录“当时为什么这样判断、做了什么、结果如何、以后能不能复用”。它不是展示成绩的地方，而是团队避免重复试错、复制有效打法的经验库。

## 页面结构

| 页面 | 用途 | 使用时机 |
| --- | --- | --- |
| [[案例库索引模板]] | 案例库索引和分类 | 新案例入库时 |
| [[增长与变现案例复盘模板]] | 成功/失败增长案例模板 | 放量、素材、变现、订阅、留存等案例 |
| [[重大异常复盘模板]] | 重大异常复盘模板 | 收入、花费、版本、数据、权限异常 |
| [[复盘会议流程]] | 复盘会议 SOP | 重大动作或异常结束后 |
| [[增长打法卡片模板]] | 可复用打法卡片模板 | 案例验证后可复用时 |

## 推荐分类

| 分类 | 入库标准 | 典型输出 |
| --- | --- | --- |
| 成功放量案例 | 花费扩大后回收和利润可接受 | 放量条件、国家/渠道边界 |
| 失败投放案例 | 高花费低回收或归因误判 | 暂停条件、预警信号 |
| 素材 winner 案例 | 素材在足够样本下持续贡献有效转化 | hook、first frame、CTA、适用人群 |
| 变现实验案例 | 广告配置、频控、场景或 mediation 调整有明确结果 | 配置前后对比、风险边界 |
| iOS 订阅增长案例 | Paywall、价格、权益、入口影响订阅表现 | 订阅漏斗和退款边界 |
| 留存提升案例 | 激活、核心路径、Push、活动改善留存 | 可复用产品/运营机制 |
| 重大异常复盘 | 收入、花费、数据、版本或权限造成明显影响 | 根因、防复发机制 |
| 竞品启发案例 | 竞品策略可转化为实验或素材方向 | 实验假设和执行清单 |

## 入库标准

- 必须写绝对日期、产品、Project ID、国家、渠道或版本范围。
- 必须写当时数据，而不是事后只写结论。
- 必须区分“可复用经验”和“不可复用边界”。
- 失败案例必须保留，且不能只写“执行不到位”。
- 可复用打法需要沉淀成 [[增长打法卡片模板]]。

## 复盘质量检查

| 检查项 | 合格标准 |
| --- | --- |
| 背景 | 能看懂当时为什么要做这个动作 |
| 数据 | 能复现核心判断，不依赖口头描述 |
| 动作 | 明确对象、Owner、时间和执行方式 |
| 结果 | 有对比周期和指标变化 |
| 根因 | 区分直接原因、深层原因和外部条件 |
| 边界 | 写清哪些产品、国家、渠道不能照搬 |
| 后续 | 有 SOP、模板、监控或动作规则更新 |
