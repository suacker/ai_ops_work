# 公司出海 App 增长运营知识库

> 面向出海 App 公司经营管理、买量增长、产品变现、产品经理、产品运营、BI 与研发协作。默认用于飞书 Wiki 迁移，也可在本仓库作为公司经营知识库维护。

## 1. 知识库定位

本知识库不是资料归档目录，而是公司增长决策系统。所有页面最终都要服务 4 类问题：

| 问题 | 页面应回答 | 不接受的写法 |
| --- | --- | --- |
| 经营是否健康 | 收入、花费、利润、ROI、留存、订阅、风险产品状态 | 只贴后台截图 |
| 钱该怎么投 | 国家、渠道、Campaign、素材、预算动作和复查窗口 | 只说“继续观察” |
| 产品该怎么改 | 漏斗、留存、变现、订阅、商店页和版本动作 | 只记录需求列表 |
| 经验如何复用 | 当时数据、判断依据、动作、结果、适用边界 | 只写成功故事 |

知识库必须沉淀可执行决策：放量、观察、降量、暂停候选、素材刷新、归因修复、变现配置修复、Paywall 优化、版本回滚、Push 调整、活动复盘或继续验证。

## 2. 公司业务背景

| 维度 | 当前公司背景 | 知识库要求 |
| --- | --- | --- |
| 产品类型 | 出海工具类、内容类、健康类等 App | 按 Project ID 和产品名统一索引 |
| 商业模式 | IAA、IAP、IAA + IAP 混合变现 | 报告必须标明变现类型，避免混合口径误判 |
| 买量渠道 | Google Ads、Facebook / Meta Ads 为主 | 平台数据只做上下文，预算动作以 Snowball/BI 回收质量为准 |
| 广告变现 | AdMob、AppLovin 聚合为主 | 收入异常要同时拆 DAU、展示、eCPM、IPU、广告格式和 mediation source |
| IAP/订阅 | iOS 订阅型产品偏多 | 单独看试用、首购、续订、退款和订阅 cohort LTV |

## 5. 公司级经营机制

| 机制 | 频率 | 主持人 | 输入 | 输出 |
| --- | --- | --- | --- | --- |
| 日经营扫读 | 工作日 | 增长负责人 | 前一完整自然日收入、花费、利润、异常 | P0 异常、当天检查项 |
| 周度经营复盘 | 每周 | 增长运营总监 | 上一完整 ISO 周经营数据、专题报告 | 本周 P0/P1/P2 动作看板 |
| 月度指标治理 | 每月 | BI 负责人 | 指标字典、数据源冲突、口径变更 | 指标 Owner、口径版本、待修复项 |
| 版本/实验评审 | 每版本/实验 | 产品负责人 | PRD、实验设计、上线数据 | 继续、扩大、回滚、补实验 |
| 重大异常复盘 | 异常结束后 3 个工作日内 | 事故 Owner | 时间线、影响、根因、修复动作 | 防复发机制和监控补丁 |

## 6. 页面质量标准

所有经营、分析和复盘页面必须包含：

| 模块 | 要求 |
| --- | --- |
| 标题与范围 | 写清产品、Project ID、绝对日期、负责人 |
| 数据源 | 写清 Snowball/BI、平台后台、广告后台、商店后台或人工记录 |
| 结论 | 控制在 5 行以内，必须能被管理层快速读懂 |
| 证据 | 核心 KPI 表、维度下钻表、异常截图或报告链接 |
| 动作 | 使用 P0/P1/P2，写负责人、截止时间、复查指标 |
| 边界 | 写清样本不足、数据缺失、预测值、不可判断范围 |
| 复盘 | 完成后沉淀结果，成功和失败都要入库 |

## 8. 维护原则

- 所有结论写绝对日期，不写“最近”“本周”这类不可追溯表述。
- 买量、变现、订阅、产品实验页面必须写清数据源和口径。
- 预算、暂停、出价、创建广告等平台写操作必须有明确授权和二次确认。
- 数据不足时明确写“数据缺失”“样本不足”或“低置信度”，不要隐藏。
- 飞书 Wiki 中的多维表格适合做索引和状态流转，Markdown 适合做结论、SOP 和复盘。
- 经验页必须写适用边界，避免把单一产品、单一国家、单一渠道经验误用到全公司。
