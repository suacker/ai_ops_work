# 02 产品变现

## 定位

产品变现空间用于沉淀广告变现、IAP、iOS 订阅、价格策略、Paywall、收入异常和变现实验。

本目录面向出海 App 团队的实际运营决策，不沉淀实时数据，只沉淀方法、字段、模板、判断规则和可复用 SOP。所有结论页迁移到飞书 Wiki 时，应能直接指导变现、产品、投放、数据和客户端团队执行。

## 使用原则

1. 先拆业务模型，再拆指标。IAA、IAP、订阅和混合变现不能混成一个总收入结论。
2. 广告变现必须拆到广告格式、广告场景、mediation source、国家、平台。只看总 eCPM 或总 revenue 不足以决策。
3. 订阅产品必须拆试用、首购、续订、取消、退款、LTV 和买量 cohort 回收。只看首购收入会误判长期质量。
4. Paywall 和价格实验必须同时定义主指标与护栏指标。短期收入提升不能以退款、留存、投诉和投放回收恶化为代价。
5. 混合变现产品必须明确广告和付费的优先级。广告展示策略要保护高付费倾向用户和订阅权益。
6. 所有页面只写方法与模板。需要真实数值时，回到 Snowball、AdMob、AppLovin MAX、App Store Connect 或 BI 查询。

## 变现模型与核心问题

| 模型 | 核心目标 | 重点指标 | 常见误判 |
| --- | --- | --- | --- |
| IAA | 在不伤留存和付费潜力的前提下提升广告收入 | ads revenue、ARPDAU、eCPM、IPU、展示率、填充率、show rate、ad ARPDAU | 只看 eCPM，把收入下降误判为竞价问题，忽略 DAU、场景曝光和格式结构 |
| IAP | 提升一次性购买转化和 ARPPU | purchase rate、ARPPU、订单数、退款率、权益使用率 | 只看订单数，忽略入口曝光、权益理解和退款 |
| iOS 订阅 | 提升试用转正、续订和长期 LTV | trial start、trial conversion、first purchase、renewal、churn、refund、LTV、cohort ROI | 只看首购收入，忽略试用质量、续订衰减和买量 cohort 回收 |
| 混合变现 | 在广告和付费之间做用户分层收益最大化 | IAA/IAP 收入占比、paywall CVR、ad ARPDAU、payer ads exposure、LTV | 对所有用户使用同一广告频控，导致广告挤压付费或付费屏蔽过度损失广告 |

## 通用字段字典

| 维度/指标 | 说明 | 必拆场景 |
| --- | --- | --- |
| `platform` | iOS / Android | 所有收入分析 |
| `country` | 国家或地区 | 买量国家、广告 eCPM、订阅价格分析 |
| `app_version` | App 版本 | 异常、实验、体验护栏 |
| `network` / `mediation_source` | AdMob Network、AppLovin、Liftoff、Meta Audience Network 等 | 广告收入、填充、竞价 |
| `ad_format` | AppOpen、Interstitial、RewardedVideo、Banner、Native | 所有广告分析 |
| `ad_scene` | 触发广告的产品场景，例如启动、结果页、内容详情页、工具完成页 | 广告策略和体验问题 |
| `ad_unit` | 平台侧广告位或聚合单元 | mediation 调整和事故定位 |
| `requests` | 广告请求数 | 填充和 SDK 问题 |
| `matched_requests` / `fills` | 成功填充数 | fill rate 分析 |
| `impressions` | 展示数 | 收入、show rate、IPU |
| `show_rate` | impressions / requests 或 impressions / fills，需注明口径 | 场景触发链路 |
| `eCPM` | Revenue / impressions * 1000 | 价格、竞价、水位 |
| `IPU` | impressions / DAU 或 impressions / active users | 体验和展示机会 |
| `ARPDAU` | Revenue / DAU | 总体变现效率 |
| `paywall_exposure` | Paywall 曝光用户数 | IAP / 订阅漏斗 |
| `trial_start` | 试用开始 | iOS 订阅 |
| `trial_conversion` | 试用转正率 | iOS 订阅 |
| `first_purchase` | 首次付费 | IAP / 订阅 |
| `renewal` | 续订 | 订阅健康 |
| `refund` | 退款 | 价格、承诺、产品质量 |
| `cohort_ltv` | 按安装 cohort 归因的长期收入 | 买量回收 |

## 通用决策边界

| 状态 | 定义 | 允许动作 |
| --- | --- | --- |
| `ready` | 核心维度齐全，样本充足，数据源一致或差异已解释 | 可给 P0/P1/P2 明确动作 |
| `partial` | 少数维度缺失，方向可信但不能精确归因 | 只给 P1/P2 或低风险验证动作 |
| `blocked` | 主数据缺失、口径冲突无法解释、样本不足 | 只给数据修复和补埋点动作 |
| `guardrail_breach` | 收入提升伴随留存、退款、投诉、崩溃或投放回收恶化 | 暂停扩量或回滚候选，先定位伤害来源 |

## 变现分析最小输出

每份变现分析至少包含：

1. 产品、平台、国家、日期范围、数据源和主口径。
2. 5 行以内的总体结论。
3. 按广告格式、广告场景、mediation source、国家拆解的主表。
4. iOS 订阅产品需要补充试用、首购、续订、退款、LTV 和买量 cohort。
5. 异常原因拆分：流量、请求、填充、展示、价格、场景、版本、配置、实验、平台回传。
6. P0/P1/P2 动作建议，并标注负责人、复查日期和回滚条件。
7. 置信度说明和不能判断的范围。

## 禁止事项

- 不要编造实时收入、eCPM、ROI 或 LTV。
- 不要把 `null`、`N/A`、`-1` 或未回传当作 0。
- 不要用广告平台总收入单独判断业务净损失；必须看其他 source 补偿和整体收入。
- 不要用短期 ARPDAU 提升直接宣布实验成功；必须检查留存、退款、投诉和买量 cohort。
- 不要在未确认具体对象和授权前执行广告平台写操作，例如改水位、暂停广告位、调整价格或发布远程配置。
