# 广告变现异常分析模板

## 适用场景

用于处理广告收入、ARPDAU、eCPM、填充率、展示率、IPU 或某个广告 source 突然波动的问题。目标不是只说明收入涨跌，而是定位损失发生在流量、请求、填充、展示、价格、场景、版本、配置还是平台回传，并给出可执行动作。

本模板不填写实时数据。执行时必须从 Snowball、AdMob、AppLovin MAX 或 BI 拉取真实数据后替换占位。

## 排查顺序

| 顺序 | 问题 | 必查字段 | 结论写法 |
| --- | --- | --- | --- |
| 1 | 是真实业务异常还是数据延迟 | revenue、impressions、requests、平台侧更新时间、Snowball 入库时间 | `确认/未确认数据完整性` |
| 2 | 异常发生在哪个产品面 | platform、country、app_version、ad_format、ad_scene、mediation_source | `异常集中在...` |
| 3 | 收入下降来自流量还是效率 | DAU、DNU、sessions、ad eligible users、ARPDAU | `流量变化贡献...` |
| 4 | 请求链路是否正常 | requests、request users、request per user、SDK error | `请求缺口/无请求缺口` |
| 5 | 填充是否异常 | fills、fill rate、no fill、timeout、adapter error | `填充异常来自...` |
| 6 | 展示是否异常 | impressions、show rate、IPU、scene entry、ad closed/error | `展示机会或展示转化变化...` |
| 7 | 价格是否异常 | eCPM、bidder eCPM、waterfall floor、bid response | `价格下降来自...` |
| 8 | 结构是否变化 | format mix、scene mix、country mix、source mix | `收入结构迁移...` |
| 9 | 是否有配置/版本/实验变更 | remote config、release、A/B group、mediation change log | `与变更...时间一致/不一致` |
| 10 | 是否影响用户体验或付费 | retention、crash、complaint、paywall CVR、purchase rate | `存在/不存在护栏风险` |

口径说明：

- `ARPDAU = ads revenue / DAU`。
- `eCPM = ads revenue / impressions * 1000`。
- `IPU = impressions / DAU` 或 `impressions / ad active users`，必须注明分母。
- `Show Rate` 必须注明是 `impressions / requests` 还是 `impressions / fills`。

场景结论必须回答：

- 场景入口是否下降。
- 入口到请求、请求到展示是否断裂。

判断规则：

- 如果某 source 收入下降但其他 source 同场景补偿明显，优先判断为流量迁移或竞价胜出变化，不要直接判断整体业务损失。
- 如果所有 source 同时 requests 正常、fill 正常但 eCPM 下滑，优先检查国家 mix、用户质量、竞价环境和底价配置。
- 如果某 source fill 或 win rate 断崖式下降，检查 adapter、SDK 版本、账号限制、隐私授权、地域屏蔽和平台事故公告。
- 如果 Snowball 只能看到聚合 source，付款侧归因需要回到 AdMob / AppLovin 原生报表确认。

| Android | US |  |  |  |  |  |  |  |  |

国家层结论必须避免平均数误导：

- 高 eCPM 国家下降对收入影响更大，不能只按 DAU 权重排序。
- 买量国家变化会影响用户质量和广告竞价，需同步查看 cohort 或 campaign。
- iOS 与 Android 的广告权限、ATT、SDK、订阅权益不同，不能混写一个原因。

## 损失估算

| 项 | 公式 | 占位 |
| --- | --- | ---: |
| 基线广告收入 | `基线 DAU * 基线 ARPDAU` 或同星期均值 |  |
| 实际广告收入 | 异常期实际 revenue |  |
| 广告侧毛损失 | `基线广告收入 - 实际异常 source/format 收入` |  |
| 其他 source 补偿 | 异常期其他 source/format 相对基线增加额 |  |
| 业务净损失 | `广告侧毛损失 - 其他补偿` |  |
| 置信度 | high / medium / low |  |

损失估算规则：

- 优先使用 DAU 归一化基线，避免把流量波动误算成变现事故。
- 对比窗口优先选择同星期或近 7/14/28 天完整自然日。
- 不要用单日前一日差值直接估算损失，除非业务强周期性已排除。
- payment-side 损失和 business net loss 要分开写。

## 根因分类

| 分类 | 典型信号 | 需要谁处理 |
| --- | --- | --- |
| 流量变化 | DAU/DNU/session 下滑，广告效率稳定 | 增长、产品 |
| 请求断裂 | scene UV 正常但 requests 下降 | 客户端、产品 |
| 填充异常 | requests 正常但 fill rate 下降 | 变现、平台、客户端 |
| 展示异常 | fills 正常但 impressions/show rate 下降 | 客户端、产品 |
| 价格下降 | impressions 正常但 eCPM 下降 | 变现、广告平台 |
| 结构迁移 | format/source/country mix 改变 | 变现、产品、增长 |
| 配置错误 | 异常与 remote config / mediation change 同步 | 变现、客户端 |
| 版本问题 | 异常集中在新版本 | 客户端、QA |
| 数据回传 | 平台报表和 Snowball 不一致 | 数据、平台 |
